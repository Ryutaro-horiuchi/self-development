# 5.1.1 ログイン機能

## ログイン機能に対する攻撃

- SQLインジェクション攻撃によるログイン機能のバイパス
    - SQLインジェクションによる認証回避
- ログイン画面に対するパスワード試行
    - ブルートフォース攻撃(総当たり)
        - パスワードの文字列の組み合わせを全て試す方法
    - 辞書攻撃
        - パスワードに用いられやすい文字列の集合を「辞書」として用意しておき、順に試す方法
- ソーシャルエンジニアリングによるパスワード入手
    - 管理者や利用者を騙して重要情報を得る手法のこと
    - 管理者になりすましでパスワードを聞き取る方法や、利用者がパスワードを入力している場所で、画面やキーボードを覗き見して得る方法もソーシャルエンジニアリングの一種
- フィッシングによるパスワード入手

## 不正ログインを防ぐためには

### SQLインジェクションなどセキュリティバグを無くす

- SQLインジェクション
- セッションIDの固定化
- クッキーのセキュア属性不備

### パスワードを予測困難なものにする

- パスワードの覚えやすさ(ユーザーの利便性)とセキュリティ強度は相反する傾向にあるが、利用者側の工夫次第では十分な強度と使いやすさを備えたパスワードを決めることが可能

### パスワードの文字種と桁数の要件

- パスワードの組み合わせ総数 = パスワードに使用できる文字種 × 桁数であるため、どちらかを増やすだけでもパスワードの組み合わせ総数は大幅に増加する

### パスワードに関するアプリケーション要件

- 文字種や桁数を必要時ように厳しくせず、利用者が安全なパスワードをつけることを邪魔しないこと
    - 「8桁まで 英数字のみ」は厳しすぎる ❌
    - 「128桁以下、ASCII文字全て」⭕️

### 積極的なパスワードポリシーのチェック

- 桁数、文字種に関するものに加え、ユーザーIDと同じパスワードを使っていないか、ありがちな単語を禁止するなどのチェックを登録時に行うと良い
- やりすぎだと帰って安全なパスワードをつける利用者のモチベーションを下げるという指摘もある
    - やりすぎ例「英字・数字・記号をそれぞれ1字は入れる」
    - NISTが発行するガイドラインでも「異なる文字種を組み合わせる規則や的変更の強制は推奨しない」という内容に変更されている

# 5.1.2 パスワード認証を狙った攻撃への対策

## 基本的なアカウントロック

- 最も基本的なアカウントロック
    
    ```
    ・ユーザーIDごとにパスワード間違いの回数を数える
    ・パスワード間違いの回数が上限を超えると、アカウントをロックする。ロックされたアカウントは
    　ログインできなくなる
    ・アカウントロックが発生した場合はメールなどで対象利用者と管理者に通知する
    ・正常にログインした場合は、パスワード間違いのカウンタをクリアする
    ```
    
    - 間違いの回数はATMと同じ3回では少なすぎ、10回程度が推奨
- ロックされたアカウントの再有効化は以下のルールで行うと良い
    
    ```
    ・アカウントロックから30分経過した場合、自動的に再有効化される
    ・管理者が、何らかの方法で本人確認した後に再有効化する
    ```
    

## パスワード認証に対する攻撃のバリエーションと対策

### 辞書攻撃

- 使用頻度の高いパスワード候補から順に試す攻撃
- アカウントロックが対策として有効

### ジョーアカウント探索

- ジョーアカウント: ユーザーIDと同じパスワード設定しているアカウント

### リバースブルートフォース攻撃

- 通常のブルートフォース攻撃がIDを固定して、パスワードの方を入れながらログイン試行する
- リバースブルートフォース攻撃は、パスワードの方を固定して、ユーザーIDを取り替えながらログイン試行する

### パスワードスプレー攻撃

- IDもパスワードも固定せずに、少数のパスワード候補をIDを変えながら試していく攻撃

### パスワードリスト攻撃

- 攻撃対象サイトとは別のサイトから漏洩したIDとパスワードの一覧を用いてログイン試行する攻撃

### パスワードを狙った攻撃への対策

- 2段階認証への実装
    - ログイン毎に行うケースの他に、以下の場合に要求する方法もある
        - 初回ログイン時のみ2段階認証を行、そのあとは一定期間一段階の認証とする
        - 利用者の通常の使い方から外れたログイン(地域、時間帯、ブラウザなど)
        - 決済やパスワード変更などの重要な処理の際
- 積極的なパスワードチェック
- ログイン失敗率の監視
    - ログイン失敗率を監視。急速に上昇した場合は該当するリモートIPアドレスとの通信遮断などを検討

# 5.1.3. パスワードの保存方法

## 暗号化によるパスワード保護と課題

- パスワードの暗号化・復号化はライブラリが用意されているのもあり、プログラミング自体は難しくないが、以下課題
    - 安全な暗号アルゴリズムの選定
    - 暗号化・復号処理の安全な実装
    - 暗号アルゴリズムが危殆化した場合の再暗号化
    - 鍵の生成・保管
        
        → 鍵の保管方法が一番難しい。
        
        - ログインのたびに鍵が必要になるため、安全な金庫にしまっておけるものではない。
        - 鍵をWebアプリケーションから参照でき、盗難はされないという方法が難しい
        
        <aside>
        💡
        
        現実にはパスワードを(復元可能な)暗号により保護されることはあまり行わず、「メッセージダイジェスト」による保護が広く行われている
        
        </aside>
        

## メッセージダイジェストによるパスワード保護と課題

### メッセージダイジェスト

- ハッシュ関数
    - 任意の長さのデータを固定調のデータ(メッセージダイジェスト、あるいはハッシュ値)に圧縮する関数と呼ぶ
- 暗号学的ハッシュ関数
    - セキュリティ上の要件を満たすハッシュ関数
        - 要件
            - 原像計算困難性
                - ハッシュ値から元データを見つけることが現実的な時間では困難である性質
            - 第2原像計算困難性(弱衝突耐性)
                - 元データが与えられた時に、元データと同じハッシュ値を持つ別のデータを見つけることが現実的には困難であるという性質
            - 衝突困難性
                - 同じハッシュ値を持つ2つのデータを見つけることが困難である性質のこと
- メッセージダイジェストの状態で保存し、ログイン時にはメッセージダイジェストのまま照合する

### ハッシュ値から元のパスワードを解析するための手法

#### 脅威1: オフラインブルートフォース攻撃

- ハッシュ値から元データを復元することはできないが、パスワードは事情が異なる
    - 短い文字列で文字種も限られているため、総当たり的に元データを探索できる場合がある
- メッセージダイジェストから元のパスワードを探す処理は、攻撃対象サーバーに接続せずに(オフラインで)できることから、オフラインブルートフォース攻撃と呼ばれる
    - 単純なMD5やSHA-1によるハッシュ値だと、現実的な時間で元データが探索できてしまう

#### 脅威2: レインボーテーブル

- パスワードのハッシュ値をあらかじめ総当たりで表にしておき、解読の際は表の参照を行なって、高速に検索する
    - 8文字までのUS-ASCII文字全部に対応したMD5用テーブルや10文字までの英小文字と数字に対応したSHA-1用テーブルが売られている
- 対策はパスワードを長くする。とはいえ、20文字以上のパスワードを強要するのは現実的ではないため、後述するsaltという方法で対策する

#### 脅威3: ユーザDB内にパスワード辞書を作られる

1. 攻撃者がダミーのユーザーを多数登録して、ユーザーDB上にパスワード辞書を作る
2. 何らかの方法で、ユーザーDBの内容を持ち出す
3. パスワード欄を調べ、ダミーで登録したパスワードのハッシュ値と同じ値をパスワードにもつユーザーを調べる

#### 対策1: ソルト

- ハッシュの元データに追加する文字列のこと
    - ある程度の長さを確保する
    - ユーザー毎に異なるものにする
        - 多くの場合乱数を用いる
- 見かけのパスワードを長くするとともに、ユーザーごとに異なるものにすることでパスワードが同じでも異なるハッシュ値が生成される

#### 対策2: ストレッチング

- ソルトを使っても、ブルートフォース攻撃の脅威は残る
- ブルートフォース攻撃に対抗するために、ハッシュ計算の速度を遅くする必要がある
    - ハッシュ計算を繰り返し行うストレッチング
    - パスワード保存に適した専用の「遅い」ハッシュ関数を用いること
