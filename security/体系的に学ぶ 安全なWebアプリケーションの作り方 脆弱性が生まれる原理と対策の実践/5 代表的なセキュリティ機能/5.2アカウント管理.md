# 5.2.1 ユーザー登録

ユーザー登録における機能的な注意点について取り上げる

## メールアドレスの受信確認

- メールアドレスを登録・変更する際には、登録されたメールアドレスに送信されるメールを利用者自身が受信できることを確認するべき
    - パスワード変更などのメールが誤って別人に届くと、セキュリティの問題につながるため
    - 方法
        - A. メールにトークン付きURLを添付して、そのURLから処理を継続する
            - ユーザーの操作性は良い
            - 利用者にメールのURLを閲覧させることは好ましくない
                - → この観点で本書では非推奨
            - 実装はやや面倒
        - B. メールアドレスを入力した後、トークン(確認番号)入力画面に遷移する。トークンは、指定したメールアドレスに送信される
            - 利用者にメールのURLを閲覧させることを避けられる
            - 確認番号を入力する操作が利用者にはやや面倒
            - 実装は容易

## ユーザーIDの重複防止

- Ex.
    - パスワードを違えば同じIDで登録できるサイト
    - ユーザーIDに一意制約をつけられないサイト
        - ユーザーIDの列にUNIQUE制約をつけられない
            - ユーザーの削除を論理削除にしているため、制約はつけられないとのこと
- リスク
    - 同一ユーザーIDで別のアカウントが作成できると、誤って別ユーザーとしてログインできる可能性が出てしまう
- 対策
    - 一意制約をつけることが望ましい
    - つけられない場合は、アプリケーション側で排他制御など細心の注意を払って、同じユーザーIDで登録できないようにする

## ユーザーの自動登録への対処

- 自動操作によって大量に新規ユーザーを作成される場合がある
- 自動登録が想定される、あるいはすでに被害が出ている場合はCAPTCHA(キャプチャ)という仕組みで対策する方法がある

### CAPTCHAによる自動対策

- Webサイトを操作しているユーザーが確かに人間であること
- コンピューターの自動応答でないことを確かめるために、わざと文字を歪めさせた画像を表示させ、利用者に文字列を入力してもらう方法
- Google社のreCAPTCHAが広く使われている
