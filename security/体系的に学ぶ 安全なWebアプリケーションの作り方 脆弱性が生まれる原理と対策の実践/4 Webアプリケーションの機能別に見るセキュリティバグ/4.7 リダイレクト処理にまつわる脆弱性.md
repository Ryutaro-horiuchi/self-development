# 4.7.1 オープンリダイレクト

## 概要

- Webアプリケーションの中には、パラメーターにより指定したURLにリダイレクトできる機能を備えたものがある
    - Ex. Googleは以下のURLでログインすると、continueパラメーターで指定したURLにリダイレクトする
        
        ```
        https://www.google.com/accounts/ServiceLogin?continue=https://mail.google.com/mail
        ```
        
    - 任意のドメインにリダイレクトできる脆弱性をオープンリダイレクト脆弱性と呼ぶ
- リダイレクトを通じてフィッシングという詐欺に悪用される可能性がある
    - フィッシング: 著名なサイトに偽装したサイト利用者を誘導して、個人情報などを入力させる手口

## 攻撃手法と影響

- ログインページ47-001.phpは、ログイン後urlパラメーターで指定した画面に遷移することができる
- 攻撃者が罠サイト47-900.phpを用意し、下記のURLにてメールやSNSなどで閲覧するように心がける
    
    ```
    http://example.jp/47/47-001.php?url=http://trap.example.com/47/47-900.php
    ```
    
    - 47-900.php
        
        !image.png
        
        - 罠サイトにて上記のような画面を表示し再度入力させる
        - 利用者の多くはドメイン名が本物であり、HTTPSの場合も証明書エラーが表示されないので、入力されやすい

## 脆弱性が生まれる原因

```
- リダイレクト先のURLを外部から指定できる
- リダイレクト先のドメイン名のチェックがない
```

上記はand条件であるため、どちらか一方を当てはまらないようにすれば、脆弱性がなくなる

## 対策

- リダイレクト先のURLを固定する
    - リダイレクトを外部からの指定ではなく、固定のURLに遷移することを検討
- リダイレクト先のURLを直接指定せず番号指定
    - ページ番号の形で指定する。ページ番号とURLの対応表はDBなどで管理
- リダイレクト先のドメイン名をチェックする
- クッションページを活用する
    - 外部ドメインに対しては直接リンクせず、外部のドメインに遷移することを利用者に告知することにより、フィッシングの被害を防止する

# 4.7.2 HTTPヘッダ・インジェクション

## 概要

- リダイレクトやクッキー発行など、入力パラメーターを元にHTTPレスポンスヘッダを出力する際に発生する脆弱性
- レスポンスヘッダを出力する際のパラメーター中に改行を挿入する攻撃によって、被害者のブラウザで以下のいずれか、またはその両方が引き起こされる
    - 任意のレスポンスヘッダーの追加
    - レスポンスボディの偽造

## 攻撃手法と影響

Ex. PerlによるCGIプログラム。urlクエリパラメーターを受け取り、urlの示すURLにリダイレクトする

- プログラムとしては、urlクエリパラメーターの値をLocationヘッダとしてセットするもの
- 下記のURLにてアクセスすると、罠サイトページに遷移されてしまう
    
    `http://example.jp/47/47-020.cgi?url=http://example.jp/%0D%0ALocation:+http://trap.example.com/47/47-900.php`
    
    - `%0D%0A`が改行であり、CGIプログラムではLocationヘッダが下記の2行になる
        
        ```
        Location: http://example.jp
        Location: http://trap.example.com/47/47-900.php
        ```
        
    - Apacheがプログラムから受け取ったLocationヘッダが複数あると、Apacheは最後のLocationヘッダのみをレスポンスとして返すため、本来のリダイレクト先は消える
    - 最終的に返されるヘッダ
        
        ```
        Location: http://trap.example.com/47/47-900.php
        ```
        

### 任意のクッキー生成

- 下記にリクエストすると、HTTPヘッダ・インジェクションの脆弱性により、クッキーが勝手にセットされてしまう
    
    `http://example.jp/47/47-020.cgi?url=http://example.jp/47/47-003.php%0D%0ASet-Cookie:+SESSID=ABCD123`
    
    ```
    Location:http://example.jp/47/47-003.php
    Set-Cookie:SESSID=ABCD123
    ```
    

### 偽画面の表示

Ex. pageidというクエリ文字列を受け取り、そのままPAGEIDという名前のクッキーとして生成する

- 以下のURLだと、「○○銀行は破産しました」という画面が変更される
    - `http://example.jp/47/47-021.cgi?pageid=P0D%0A%0D%0A○○銀行は破産しました`
- Set-Cookieヘッダに続けて改行を2回連続で出力したことで、後のデータがレスポンスボディとみなされてしまっている

## 脆弱性が生まれる原因

- レスポンスヘッダはテキスト形式で改行を用いて1行に1つのヘッダが定義できる
- この性質を悪用して、リダイレクト先やクッキー値として使用する入力パラメーターに改行を挿入した場合に、そのままレスポンスヘッダとして出力されることがHTTPヘッダ・インジェクション脆弱性の原因

## 対策

1. 外部からのパラメーターをHTTPレスポンスヘッダとして出力しない
    - リダイレクトの場合は固定にするなど
2. 以下の両方を実行する(1が仕様として不可能な場合)
    - リダイレクトやクッキー生成を専用APIに任せる
    - ヘッダ生成するパラメーターの改行をチェックする
        - URL中の改行はエラーとする
        - クッキー値の改行はパーセントエンコードする
            - パーセントエンコーディング
