# 4.12.1 ファイルアップロードの問題の概要

- アップロード機能に対するDos攻撃
    - 大量のデータ送信によるDoS攻撃
    - 対策
        - アプリケーションの要求を満たす範囲で、アップロードファイルの容量をできるだけ小さな値に制限
        - Apacheのhttpd.confの設定でリクエストボディサイズを絞ることもできる
- アップロードされたファイルをサーバー上のスクリプトとして実行する攻撃
    - 外部からアップロードしたスクリプトファイルがWebサーバー上で実行できてしまう可能性がある
- 仕掛けを含むファイルを利用者にダウンロードさせる攻撃
    - 仕掛けを含むファイルを攻撃者がアップロードし、そのファイルを利用者に閲覧させ、利用者のPC上でJSのスクリプトの実行、マルウェア感染などが起こる

# 4.12.2 アップロードファイルによるサーバー側スクリプト実行

## 概要

- 利用者がアップロードしたファイルをWebサーバーの公開ディレクトリに保存
- 加えてファイル名の拡張子にサーバー側で実行可能なスクリプト言語の拡張子が指定できると、アップロードしたファイルをスクリプトとしてサーバー上で実行できる
- 外部からのスクリプトが実行されると、OSコマンド・インジェクションと同様の影響がある

## 攻撃手法と影響

攻撃の例

```php
<pre>
<?php
	system('/bin/cat' etc/passwd');
?>
</pre>
```

- system関数によりcatコマンドを呼び出し、`/etc/passwd`の内容を表示する

## 脆弱性が生まれる原因

- アップロードしたファイルが公開ディレクトリに保存される
- アップロード後のファイル名として、`.php、.asp、.aspx、jsp` などサーバースクリプトを示す拡張子が指定できる

## 対策

- ファイルを公開ディレクトリに保存しないか、拡張子の制限が有効
    - 拡張子だけだと対策抜けが生じる可能性があるため、ファイルを公開ディレクトリに保存しない方法を推奨

# 4.12.3 ファイルダウンロードによるクロスサイト・スクリプティング

## 概要

- 利用者がファイルをダウンロードする際に、ブラウザがファイルタイプを誤認する場合がある
    - PDFファイル中にHTMLタグが含まれていると条件によってはHTMLファイルと誤認。
    - PDFファイルに埋め込まれたJavaScriptファイルを実行するのが、ファイルダウンロードにおけるXSSである

## 攻撃手法と影響

### 前提

ファイルアップロードすると、下記URLが発行される

- Ex. 正常系
    
    ```
    http://example.jp/4c/4c-013.php?file=0b1e764c.pdf
    ```
    
    - こちらのリンクを叩くとダウンロードされる

### 攻撃

- PDFに偽装したHTMLファイルをアップロード
    
    ```
    <script>alert('XSS');</script>
    ```
    
- 発行されたURLに対し、`/a.html`というPATH_INFOを埋め込む
    
    ```
    http://example.jp/4c/4c-013.php/a.html?file=47972fc1.pdf
    ```
    
    - こちらのリンクを叩く
        - レスポンスヘッダ
            
            ```
            Content-Type: application/x-pdf
            ```
            
            - application/x-pdfという間違ったContent-Typeが指定されている
        - IE環境だとスクリプトが実行されてXSSになる
        - Firefox環境だと一度ダウンロードされてからになるため、XSSには至らない

## 脆弱性が生まれる原因

- Content-Typeの間違った指定が原因
    - ブラウザがコンテンツをHTMLと解釈してJavaScriptが実行されてしまう可能性がある

## 対策

- ファイルアップロード時とダウンロード時の対策が必要
    - アップロード時は、拡張子が許可されたものがどうか

### ファイルダウンロード時の対策

- Content-Typeを正しく設定する
- レスポンスヘッダ`X-Content-Type-Options: nosniff`を指定する
    - Content-TypeヘッダのみからContent-Typeを解釈するようになる
    - 歴史的に、IE側で使用できないContent-Typeが指定されていた場合は、URLに含まれる拡張子からファイルタイプを判定していた。
- 必要に応じてContent-Dispositionを設定する
    - ダウンロードファイルをアプリケーションで開くのではなく、ダウンロードできされすれば良い場合は`Content-Disposition: attachment`を指定するとよい
    - この場合、Content-Typeも`application/octet-stream`にすると、ファイルタイプ上もダウンロードすべきファイルという意味になる

# 4.12.4 PDFのFormCalcによるコンテンツハイジャック

## 概要

- PDFはFormCalcと呼ばれるスクリプト言語が使用でき、PDFドキュメントに埋め込むことができる
- Adobe Acrobat Readerに実装されたFormCalcにはURL関数というHTTPリクエストを呼び出し、結果を受け取ることができる機能があり、これを悪用して正規ユーザーになりすましを行う攻撃手法がある
- Adobe Acrobat Readerプラグインを備えたブラウザが影響を受けるため、主にIEが対象ブラウザになる

## 攻撃手法と影響

!IMG_1075.jpeg

- 利用者はexample.jpにログイン状態で罠ページを閲覧
- 罠ページ内にあるPDFのFormCalcスクリプトにより、example.jpにリクエストが送信される。
    - この時利用者のexample.jpのクッキーをAdobe Readerが引き継ぐ形でこのリクエストにも付与される
    - → 秘密情報が抜き取られる

## 脆弱性が生まれる原因

- Adobe Acrobat Readerのセキュリティ上好ましくない仕様を悪用したもの

## 対策

- PDFファイルはブラウザ内で開かずダウンロードを強制する
    - IE環境ではレスポンスヘッダに、`X-Download-Options: noopen`を使用すると、ファイルを開くのボタンが非表示になる
- PDFをobject要素やembed要素では開けない仕組みを実装する
    - ファイルダウンロードをPOSTリクエストに限定するなど
