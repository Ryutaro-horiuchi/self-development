# 4.8.1 クッキーの不適切な利用

- 書き換えられると困る情報をクッキーに保存すると脆弱性の原因になる
    - ユーザーIDや権限情報など
- セッション管理機構を用いて、セッションIDのみをクッキーに保存、データ自体はWebサーバーのメモリやファイル、 DBなどに保存するのが良い

# 4.8.2 クッキーのセキュア属性不備

## 概要

- クッキーのセキュア属性
    - HTTPS通信のみブラウザからサーバーに送信される
    - HTTPS通信を利用していても、セキュア属性のついていないクッキーは平文で送信される場合があり、盗聴される場合がある

## 攻撃手法と影響

1. セキュア属性のついていないクッキーを返すページにアクセス
2. 下記のページにアクセス
    
    ```
    <img src="http://wasbook.org:443/" width="0" height="0">
    ```
    
    - `http://wasbook.org:443/`のアクセス時に、1のクッキーがリクエストヘッダーに含まれていることがわかる

イメージ

p261

## 脆弱性が生まれる原因

- 直接的な原因はセキュア属性をつけていないだけだが、セキュア属性をつけない主な原因は以下2種類
    - 開発者がセキュア属性について知らない
    - セキュア属性をつけるとアプリケーションが動かなくなる

### クッキーにセキュア属性がつけられないアプリケーション

- HTTPとHTTPSの両方を使うショッピングサイトなどのWebアプリケーション
    - カタログから商品を選ぶ際はHTTP通信、商品選びが終わってからの決済はHTTPSを利用

## 対策

- クッキーにセキュア属性をつけること
- トークンを用いた対策(セッションIDを保持するクッキーにセキュア属性をつけられない場合)
    - 別でトークンを保持するクッキーにセキュア属性を付与する
        - セッションIDが盗聴された場合でも、HTTPSページを閲覧する際に、第三者では知り得ないトークンが必要であることから、安全性が確保される

## セキュア属性以外の属性値に関する注意

- Domain 属性
    - デフォルト状態が最も安全。指定する場合は複数のサーバーでクッキーを共有する場合になるが、通常セッションIDを複数バーガー間で共有する意味はない
- Expires属性
    - セッションIDのクッキーには、Expires属性をつけず、ブラウザ終了と同時にクッキーが削除される状態にする
- HttpOnly属性
    - JSから参照できなくする。セッションIDをJavaScriptから参照する意味はないのでつけることを推奨
