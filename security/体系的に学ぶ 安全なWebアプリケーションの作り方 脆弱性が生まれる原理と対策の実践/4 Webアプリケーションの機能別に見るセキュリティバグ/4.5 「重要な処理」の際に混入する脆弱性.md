# 4.5.1 クロスサイト・リクエストフォージェリ(CSRF)

- 「重要な処理」の受付に際しては、利用者の意図したリクエストであることを確認する必要がある
    - この確認処理が抜けていると、罠のサイトを閲覧しただけで、利用者のブラウザから勝手に「重要な処理」を実行させられることがある
        
        → CSRF脆弱性とよび、悪用する攻撃をCSRF攻撃と言う
        
- CSRF脆弱性は被害者の個人情報は盗むことはできない。アプリケーションの「重要な処理」の悪用に限られる

## 攻撃手法と影響

### 入力-実行パターンのCSRF攻撃

- パスワード変更の画面を題材とする
    
    ![image.png](attachment:26d43d2e-14fd-4b9e-bbd6-3b66a61ae54c:image.png)
    
    こちらでパスワードを変更する際には以下3つの要件
    
    1. POSTメソッドで変更エンドポイントである45-003.phpがリクエストされること
    2. ログイン状態であること
    3. POSTパラメーターがpwdとしてパスワード指定されていること
- 上記条件を満たしたリクエストを送信できればCSRF攻撃となる。アクセス時にCSRF攻撃をするHTMLファイル例
    
    ```html
    <body onload="document.forms[0].submit()">
    <form action="http://example.jp/45/45-003.php" method="POST">
    <input type="hidden" name="pwd" value="cracked">
    </form>
    </body>
    ```
    
    - 利用者がexample.jpにログインしていれば、　閲覧するだけで新しいパスワード`cracked`がPOSTメソッドにより送信される
    - 実際の攻撃では罠サイト上で脆弱性のあるサイトを見えないiframeで表示して仕掛ける

### CSRF攻撃と反射型XSSの比較

- CSRFはリクエストに対するサーバー側の処理を悪用するもの
    - 悪用内容はサーバー側で用意された処理に限定される
- XSSの場合、リクエストに含まれた悪意あるスクリプトがおうむ返しにレスポンスとして返され、それがブラウザ上で実行されることで攻撃が起こる
    - ブラウザ上でできることはなんでも悪用可能

## 脆弱性が生まれる原因

1. form要素のaction属性にはどのドメインのURLでも指定できる
2. クッキーに保管されたIDは対象サイトに自動的に送信される

### 利用者の意図したHTTPリクエストとCSRF攻撃によるHTTPリクエストの違い

- Refererヘッダのみ異なっている
- Referer以外はクッキーの値も含めて同じ

## 対策

### CSRF対策の必要なページを区別する

- Webアプリケーションへの入り口は一箇所とは限らず、検索エンジンやソーシャルブックマーク、そのほかのリンク等からWebアプリケーション上の様々なページにリンクされていることが一般的であり、これら全てを対策するのは好ましくない
- ECサイトにおける物品購入、パスワード変更などの重要な画面(処理)にCSRF対策を施すべき
    
    基本設計工程で画面遷移図を作成 → CSRF対策の必要なページにマークする
    

### 正規利用者の意図したリクエストであることを確認する

|  | トークンの埋め込み | パスワード再入力 | Referer確認 |
| --- | --- | --- | --- |
| 開発工数 | 中 | 中 | 小 |
| 利用者への影響 | なし | パスワード入力の手間が増える | Refererをオフにしている場合、利用者が使えなくなる |
| 推奨される利用シーン | 最も一般的な対策方法であり、推奨 | なりすまし対策や、確認を強く求めるような要件の場合 | 利用者の環境を限定できる |

## 保険的対策

- 重要な処理の実行後に対象利用者の登録済みメールアドレスに対して、処理内容の通知メールを送信することを推奨
- 万が一CSRF攻撃を受けた際に利用者が早期に気づき、被害を最小限に止めることができる可能性がある

# 4.5.2 クリックジャッキング

iframe要素とCSSを巧妙に利用することで、透明にした攻撃対象ページと罠のサイトを重ね合わせ、利用者が気付かないうちに攻撃対象サイトでのクリックを誘導する攻撃手法

### 例 掲示板(Xライク)のウェブインテント機能

- クエリー文字列から投稿内容を指定して、投稿フォームにあらかじめ指定できる
    
    []()
    

### クリックジャッキング

- 罠ページ上で上記の攻撃対象フォームをiframe要素を用いて重ねる。
    - z-indexを用いて罠画像を奥、攻撃対象フォームを手前
    - 攻撃対象フォームを透明に指定する
- 利用者から見ると罠サイトだけが見えるが、ボタンを押そうとすると「wasbook小学校を襲撃します」が投稿される
    
    []()
    

## 対策

- frameおよびiframeでの参照を制限する`X-Frame-Options`というレスポンスヘッダを対応する
    - 値はDENY(拒否)か、SAMEORIGIN(同一生成元に限り許可)のいずれかの値を取る
    - DENYを指定したレスポンスはiframeで表示されなくなる
    - SAMEORIGINはアドレスバーに表示されたオリジンと同じオリジンである場合のみ表示される
