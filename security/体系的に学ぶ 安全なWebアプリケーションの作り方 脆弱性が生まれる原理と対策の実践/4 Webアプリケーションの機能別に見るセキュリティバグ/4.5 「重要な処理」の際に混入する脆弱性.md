# 4.5.1 クロスサイト・リクエストフォージェリ(CSRF)

- 「重要な処理」の受付に際しては、利用者の意図したリクエストであることを確認する必要がある
    - この確認処理が抜けていると、罠のサイトを閲覧しただけで、利用者のブラウザから勝手に「重要な処理」を実行させられることがある
        
        → CSRF脆弱性とよび、悪用する攻撃をCSRF攻撃と言う
        
- CSRF脆弱性は被害者の個人情報は盗むことはできない。アプリケーションの「重要な処理」の悪用に限られる

## 攻撃手法と影響

### 入力-実行パターンのCSRF攻撃

- パスワード変更の画面を題材とする
    
    ![image.png](attachment:26d43d2e-14fd-4b9e-bbd6-3b66a61ae54c:image.png)
    
    こちらでパスワードを変更する際には以下3つの要件
    
    1. POSTメソッドで変更エンドポイントである45-003.phpがリクエストされること
    2. ログイン状態であること
    3. POSTパラメーターがpwdとしてパスワード指定されていること
- 上記条件を満たしたリクエストを送信できればCSRF攻撃となる。アクセス時にCSRF攻撃をするHTMLファイル例
    
    ```html
    <body onload="document.forms[0].submit()">
    <form action="http://example.jp/45/45-003.php" method="POST">
    <input type="hidden" name="pwd" value="cracked">
    </form>
    </body>
    ```
    
    - 利用者がexample.jpにログインしていれば、　閲覧するだけで新しいパスワード`cracked`がPOSTメソッドにより送信される
    - 実際の攻撃では罠サイト上で脆弱性のあるサイトを見えないiframeで表示して仕掛ける
