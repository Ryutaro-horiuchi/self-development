# 4.4.1 SQLインジェクション

- SQLの呼び出し方に不備がある場合に発生する脆弱性
- DB内の全ての情報が外部から盗まれたり、内容を書き換えられたりする非常に影響の大きな脆弱性であるため、SQLインジェクションが絶対に混入しないようなアプローチが求められる

### SQLインジェクションによる認証回避

- 脆弱性コード
    
    ```php
    <?php
    	$id = @$_POST['id'];
    	$pwd = @$_POST['pwd'];
    	try {
    		$sql = "SELECT * FROM books WHERE id = '$id' AND PWD = '$pwd'";
    		$ps = $db->query($sql;)
    ?>
    <!-- SELECTした業が存在する場合ログイン成功 -->
    ```
    
- 攻撃
    
    ![image.png](attachment:1a08a799-6457-410f-97ce-78d9d9646929:image.png)
    
    - `' OR 'a'='a`
        - ログイン成功してしまう
    - 以下のようにSQLが組み立てられる
        
        ```sql
        SELECT * FROM books WHERE id = 'yamada' AND PWD = '' OR 'a'='a'
        ```
        
        - 末尾にOR a = 'a’ が追加されたために常にWHERE句が成立する状態となっている

## 脆弱性が生まれる原因

### 文字列リテラルの問題

- SQLの標準企画では、文字列リテラルはシングルクォートで囲む
- 文字列リテラル中にシングルクォートを含めたい場合は、シングルクォートを重ねる
    - Ex. 「O’Reilly」 → 「’O`’’`Reilly’」
- SQL脆弱性が見られるプログラムでは、シングルクォートを重ねる処理が抜けているため、次のようなSQL文が組み立てられる
    
    ```sql
    SELECT * FROM books WHERE author = 'O'Reilly';
    ```
    
    - `author = 'O'Reilly';`について、「Reilly’」がはみ出している部分であり、SQL文として意味をもたないため、エラーとなる
        
        → これの代わりにSQL文として意味を持つ文字列を挿入するとSQLインジェクション攻撃が可能になる
        

## SQLインジェクション対策

1. プレースホルダによりSQL文を組み立てる
    
    ```sql
    SELECT * FROM books WHERE author = ? ORDER BY id;
    ```
    
    - プレースホルダーには、静的プレースホルダーと動的プレースホルダーがある。
        - 静的プレースホルダー: 値のバインドをDBエンジン側で行う
            
            → 原理的にこちらの方がSQLインジェクションの可能性がない
            
        - 動的プレースホルダー: 値のバインドをアプリケーション側のライブラリ内で行う
2. アプリケーション側でSQL文を組み立てる際に、リテラルを正しく構成するなど、SQL文が変更されないようにする
- 2は難しいため、1を推奨

### SQLインジェクションの保険的対策

- 詳細なエラーメッセージの抑止
    - SQLインジェクション攻撃によりDBの内容を盗み出す手法の人に、[エラーメッセージの利用がある](https://www.notion.so/4-4-SQL-36593db0555480d9ad95e09b9f506c89?pvs=21)
    - 詳細なエラーメッセージを抑止することで、被害に遭いにくくすることができる
- 入力値の妥当性検証
    - アプリケーションの要件によるが、入力値検証を行うと結果として脆弱性対策につながる
- DBの権限設定
    - アプリケーションが利用するDBユーザーに対しては、アプリケーションの機能を実現するために必要最小限な権限のみ与えておけば、万一SQLインジェクション攻撃を受けても、被害を最小限にとどめられる
        - Ex. 商品情報を表示するだけのアプリケーションであれば、商品テーブルに対する書き込み権限は与えないなど
