# 4.17.1 DOM Based XSS

## 概要

- サーバー側のプログラム不備だけではなく、JSによる処理の不備が原因でXSSとなる場合がある
    
    → `DOM Based XSS` と呼ばれている
    

## 攻撃手法と影響

### innerHTMLによるDOM Based CSS

- 前提
    - URLの#以下の部分(フラグメント識別子やハッシュなどと呼ばれる)に応じて表示変化させるアプリケーション
    
    ```jsx
    <body>
    <script>
    window.addEventListener("hashchange", chghash, false);
    window.addEventListener("load", chghash, false);
    
    function chghash() {
      var hash = window.location.hash;
      var color = document.getElementById("color");
      color.innerHTML = decodeURIComponent(window.location.hash.slice(1));
    }
    </script>
    <a href="#赤">赤</a>
    <a href="#緑">緑</a>
    <a href="#青">青</a>
    <p id="color">赤</p>
    
    </body>
    ```
    
    - window.location.hash
        - URL のフラグメント識別子`'#'` を含む文字列を取得する
- 攻撃
    
    ```
    http://example.jp/4h/4h-001.html#<img src=/% onerror=alert(1)>
    ```
    
    → 1がalertで表示される
    

### javascriptスキームによるXSS

- 前提コード
    
    ```jsx
    function go() {
      var url = location.hash.slice(1);
      location.href = url;
    )
    ```
    
- 正常系
    - `http://example.jp/4h/4h-006.html#4h-007.html`
- 異常系
    - `http://example.jp/4h/4h-006.html#javascript:alert(document.domain)`)
    - location.hrefに任意の文字列を指定できるとXSSの原因になる

## 脆弱性が生まれる原因

- DOM操作の際に外部から指定されたHTMLタグが有効になってしまう機能を用いている
    - `document.write()` / `document.writeIn()` / `innterHTML` / `outerHTML`
- eval などの機能を外部から指定された値を引数に実行している
- XMLHttpRequestのURLが未検証である
- location.hrefやsrc属性、href属性のURLが未検証である

## 対策

### 適切なDOM操作あるいは記号のエスケープ

- innterHTMLやdocument.writeの使用を避けて、textContentプロパティを使用する
    
    ```jsx
    // 脆弱
    color.innterHTML = decodeURLComponent(window.location.hash.slice(1));
    // 対策
    color.textContent = decodeURLComponent(window.location.hash.slice(1)); 
    ```
    

### eval、setTimeout()、Functionコンストラクタなどの引数に文字列形式で外部からの値を渡さない

- evalインジェクションの原理で任意のJavaScriptが実行されてしまう
    
    4.14.1 evalインジェクション 
    
    ```jsx
    <script>
    	var sec = location.hash.slice(1);
    	setTImeout("alert('" + sec + "秒経ちました')", sec * 1000);
    </script>
    ```
    
- setTimeoutやsetIntervalの場合は、文字列の代わりに関数リテラルやクロージャーを渡すことに解決する
    
    ```jsx
    <script>
    	var sec = location.hash.slice(1);
    	setTImeout(function() "alert('" + sec + "秒経ちました')", sec * 1000);
    </script>
    ```
    

### URLのスキームをhttpかhttpsに限定する

- location.hrefやsrc属性、href属性の値を外部からの入力値として取得する際は、スキームがhttpかhttpsであることを確認する必要がある

# 4.17.2 Webストレージの不適切な使用

## 概要

- localStorageやsessionStorageなどの高機能なストレージのこと

## Webストレージには何を保存して良いか

- JavaScriptにより読み書きできるストレージであり、JavaScriptからのアクセスは禁止することはできない
    
    → XSS脆弱性がると内容が漏洩することになる
    
- 重要なパスワードや個人情報などは直接Webストレージには保存せず、必要になるたびにサーバーに問い合わせるようにする

## 不適切な利用例

- Webストレージに秘密情報を保存していた
- 保存した情報が、XSSやpostMessageにより漏洩
    - postMessage
        - iframeや、ｗindow.openで開いたウィンドウなど、複数のウィンドウが異なるオリジンで強調して動作する環境で、メッセージやデータのやり取りを行う汎用的な仕組み
        - オリジンが想定のものなのかをアプリケーション側にて確認することで対策になる

# 4.17.4 オープンリダイレクト

- 概要
    - location.hrefに外部からの入力値を代入する際、罠サイトのログインページなどにリダイレクトさせることで情報が盗まれてしまうなどの被害につながる
- 対策
    - リダイレクト先のURLを固定、あるいは番号にて指定することで対策は可能
