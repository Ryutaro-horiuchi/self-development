# 4.16.1 JSONとJSONPの概要

## JSONPとは

- CORSができる前に同一オリジンポリシーの枠内で異なるオリジンサーバーからデータを取得する方法がいくつか考案され、その代表的な方法の一つがJSONP
- script要素を用いて外部のJavaScriptを直接実行することによりデータを取得する
    - JSON文字列そのままではscript要素で受け取れないため、関数呼び出しの形で取得

#### サンプル

- JSONPの原理 API側(api.example.net)
    
    ```php
    <?php
    	$callback = $_GET['callback'];
    	header('Content-Type': text/javascript; charset='utf-8');
    	$json = json_encode(array('time' => date('G:i')));
    	echo "$callback($json);"
    ```
    
- example.jpのサイトからJSONPでデータを取得する例
    
    ```php
    <body>
    <script>
    function display_time(obj) {
    	// $`時刻は{obj.time}です`をhtmlに表示する関数
    }
    </script>
    <script src="http://api.example.net/4g/4g-003.php?callback=display_time"></script>
    </body>
    ```

# 4.16.2 JSONエスケープの不備

## 概要

- JSON文字列生成時のエスケープ処理に不備があると、意図しないJavaScriptがJSONデータに混入する場合がある

## 攻撃手法と影響

- 前提
    - クエリパラメーターzipを受け取り、見当たらなかった場合に、zipをそのままメッセージに加えて返す
        
        ```php
        <?php
        	...
        	$json = '{"message": "郵便番号が見つかりません:' . $zip . '"}';
        	echo "callback_zip($json);" ;
        ```
        
- 下記のようなパラメーターでアクセスするとJavaScriptが実行される
    
    ```php
    http://example.jp/4g/4g-007.html#1"%2balert(document.domain)%2b"
    ```
    
    `%2b`は+ をパーセントエンコードしたもの
    
    !image.png
    

## 脆弱性が生まれる原因

- JSON文字列生成時に適切なエスケープ処理が行われない
- JSONの評価にeval関数などを用いているか、JSONPを用いている

## 対策

- 文字列連結によるJSONデータ生成ではなく、信頼できるライブラリを用いてJSONを生成する
- eval関数ではなく、JSON.parseなどの安全なAPIでJSONを解釈する

# 4.16.3 JSON直接閲覧によるXSS

## 概要

- APIが返すレスポンスデータをブラウザで直接閲覧させることにより攻撃が可能になる場合がある

## 攻撃手法と影響

- 前提
    - zipクエリパラメーターで郵便番号を受け取って住所を返すAPI
    
    ```php
    <?php
    	$zip = $_GET['zip'];
    	// 郵便番号が見つからなかった場合
    	$json = '{"message": "郵便番号が見つかりません:' . $zip . '"}';
    	echo json_encode(array("message" => "郵便番号が見つかりません:" . $zip));
    ```
    
- 攻撃
    - `http://example.jp/4g/4g-011.php?zip=<img+src=1+onerror=alert(document.domain)>`%3E)
    - JavaScriptが実行されて、domain(example.jp)が出力されてしまう
    - レスポンス
        
        ```php
        ...
        Content-Type: text/html; charset=UTF-8
        {"message":"\u90f5\u4fbf\u756a\u53f7\u304c\u898b\u3064\u304b\u308a\u307e\u305b\u3093:
        <img src=1 onerror=alert(document.domain)>"}
        ```
        

## 脆弱性が生まれる原因

- MIMEタイプの間違いが直接の原因

## 対策

- MIMEタイプを正しく設定する(必須)
- レスポンスヘッダ `X-Content-Type-Options: nosniff`を出力する
    - Content-Typeよりもファイルの拡張子および中身を優先してMIMEタイプを判定する独自仕様を打ち消すためのヘッダ
- XMLHttpRequestなどCORS対応の機能だけから呼び出せるようにする
    - `X-Requested-With: XMLHttpRequest`を付与し、不正なリクエストかチェック

# 4.16.4 JSONPのコールバック関数名によるXSS

## 概要

- JSONPはコールバック関数呼び出しの形式を取る
- 外部から指定したパラメーターにより、コールバック関数が制御であることから、コールバック関数めいによるXSSが可能になる場合がある

## 攻撃手法と影響

- 脆弱なAPI
    
    ```php
    <?php
    	$callback = $_GET['callback'];
    	$json = json_encode(array('time' => date('G:i')));
    	echo "$callback($json);";
    ```
    
    - Content-Typeを指定していない。正しいのは`text/javascript`だが、デフォルトの`text/html`で返される
- 攻撃
    
    http://api.example.net/4g/4g-015.php?callback=<script>alert(document.domain)</script>%3C/script%3E)
    
    → スクリプトが実行される
    

## 脆弱性が生まれる原因

- 外部から指定されたコールバック関数名を検証しないでそのまま表示している
- MIMEタイプをtext/javascriptとするべきところをtext/htmlとしている

## 対策

- 以下のいずれかをなくせば攻撃できなくなるが、正しいプログラムを書くという観点で以下の両方を実施すべき
    - コールバック関数名の文字種と文字数を制限する
        - JavaScriptの識別子である英数字と_のみ許容
    - MIMEタイプを正しく設定する

# 4.16.5 Web APIのCSRF

参考: 4.5.1 クロスサイト・リクエストフォージェリ(CSRF) 

## WebAPIに対するCSRF攻撃経路

- GETリクエストによる攻撃
- HTMLフォームによる攻撃
    
    POSTメソッドの送信方式である3種類(`text/plain`, `application/x-www-form-urlencoded`, `multipart/form-data`)のMIMEタイプを想定している場合や、MIMEタイプのチェックをしていない場合はHTMLフォームから攻撃できることにある
    
- クロスオリジン対応のXMLHTTPRequestによる攻撃(シンプルなリクエスト)

## 対策

- WebAPIに対するCSRF対策は、トークンによる方法が使えるのでこれを第一すべき
    - 代表的な方法
        - CSRFトークン
        - カスタムリクエストヘッダによる対策
            - プリフライトリクエストの対応を適切にしていれば、更新処理のPOSTリクエストは送信されず攻撃には至らない
            - XMLHTTPRequestにカスタムリクエストヘッダを付与し、サーバー側で確認するだけでもCSRF対策になる
    - 共通対策
        - 入力データのMIMEタイプを検証
        - CORSを適切に実装する

# 4.16.6 JSONハイジャック

## 概要

JSONデータをscript要素にて受け取る手法のことをJSONハイジャックと呼ばれる

## 攻撃手法と影響

- 古いFirefoxでは下記で4g-030.jsonから返される`[{”mail”: “alice@example.jp”}, {”mail”: “bob@example.jp” }]`がalertで表示される
    
    ```php
    <body onload="alert(x)">
    罠サイト
    <script>
    var x = "";
    Object.prototype.__defineSetter__("mail", function(v) {
      x += v + " ";
    });
    </script>
    <script src="http://example.jp/4g/4g-030.json"></script>
    
    </body>
    ```
    

## 対策

- `X-Content-Type-Options: nosniff`ヘッダの付与(強く推奨)
    - 多くのブラウザでscriptタグで読み込むコンテンツMIMEタイプを厳格にチェックし、JSONやCSVなどの読み込みを拒否する
- リクエストヘッダ`X-Requested-With: XMLHttpRequest`の確認(推奨)
    - jQueryなどのJavaScriptライブラリでは、自動的にHTTPリクエストヘッダとして`X-Requested-With: XMLHttpRequest`がつく
    - script要素からでは、上記ヘッダを追加する方法がないため、JSONハイジャックとしても有効

# 4.16.7 JSONPの不適切な利用

## 概要

- JSONPはCORS前の同一オリジンポリシーの制限をすり抜けるために考案された事情がある
- 呼び出し元を検証しないと、情報流出につながる恐れがある

## 対策

- JSONPをやめ、CORS対応のXMLHttpRequestを用いること
- やむを得ず、JSONPのAPIを利用する場合でも信頼できる組織が提供していることを確認するべき
    - API提供側に悪意があると、任意のスクリプトが実行されて、XSSと同等のリスクになる

# 4.16.8 CORSの検証不備

- 利用者側の無知や手抜きなどにより問題が起こる場合がある
- Ex.
    - オリジンとして`”*”`を指定する
        - `Access-Control-Allow-Origin: *`
        - Originヘッダを検証して、レスポンスヘッダでオリジンを明示すべき
    - オリジンのチェックを緩和してしまう

# 4.16.9 セキュリティ強化するレスポンスヘッダ

- X-Frame-Options
    - クリックジャッキング対策(4.5.2 クリックジャッキング )
    - frameやiframeの内部に表示することができなくなる
- X-Content-Type-Options
    - nosniffという形で使用する
    - MIMEタイプの解釈を厳密にすることで、誤認させるタイプの攻撃などを緩和
- X-XSS-Protection
    - 利用者がXSSフィルタの有効化・無効化していても、当該ページについてXSSフィルタの設定を上書きする
- Content-Security-Policy(CSP)
    - XSS攻撃を緩和するための機能であるが、開発者の労力が大きい
        - スクリプト、画像、、CSSなどのすべてのメディアをサイト自身のオリジンから読み込むようになる
        - インラインスクリプトが禁止されてしまう
- Strict-Transport-Security(HTTP Strict Transport Security; HSTS)
    - HTTPSでの接続を強制
