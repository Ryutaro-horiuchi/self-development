# 4.14.1 evalインジェクション

## 概要

- 文字列をスクリプトソースとして解釈して実行するevalと呼ばれる機能
- evalの利用方法に問題がある場合、外部から送り込んだスクリプトを実行される場合がある。
    - この攻撃をevalインジェクション攻撃といい、そのような攻撃を受ける脆弱性をevalインジェクション脆弱性と呼ぶ

## 攻撃手法と影響

### 脆弱なアプリケーションの説明

- 複雑なデータを文字列に変換(シリアライズ)に変換してフォーム間で受け渡しをする場合の脆弱性
    - 入力フォーム一部
        - PHPのvar_exportを使用して式の値をPHPのソースの形式で返す(シリアライズ)
        
        ```php
        ...
        $ex = var_export(array(1,2,3), true)
        $b64 = base64_encode($ex)
        ...
        ```
        
    - 出力
        - evalを用いてデシリアライズ(元のデータに戻す)を行う($b64の値がクエリパラメーターとして渡される)
            
            ```php
            <?php
            	$str = base64_decode($data)
            	eval('$a = ' . $str . ';')
            ?>
            <body>
            <?php var_dump($a); ?>
            ```
            
            !image.png
            
            → 元の配列に戻っている
            

### 攻撃

evalに対して、外部からのパラメーターで任意の式を注入できる

- Ex. 下記を注入するとする
    
    ```php
    $a 0; phpinfo();
    ```
    
    - OWASP ZAPの機能で文字列をBase64エンコード(`MDsgcGhwaW5mbygp)`
    - 攻撃
        
        ```html
        http://example.jp/4e/4e-002.php?data=MDsgcGhwaW5mbygp
        ```
        
        !image.png
        
        phpinfo関数が実行されてしまう。
        

## 脆弱性が生まれる原因

- evalを扱うこと自体がそもそも危険である
- evalに与えるパラメーターのチェックがされていない

## 対策

- eval(同等機能を含む)を使わない
- evalの引数に外部からのパラメーターを使用しない
- evalの与える外部からのパラメータを英数字に制限する

# 4.14.2 安全ではないデシリアライゼーション

## 概要

- アプリケーション内部の構造を持ったデータを保存・伝達する目的でバイト列に変換することをシリアライズ。シリアライズされたデータを復元することをデシリアライズと呼ぶ
- シリアライズされたデータが信頼できない場合、デシリアライズの処理の際に意図しないオブジェクトが生成され、任意のコードを実行されてしまう場合がある

## 攻撃手法

- Cookieにシリアライズされている値をサーバー側で受け取り、ログ出力するスクリプト
- シリアライズされた値を検証せずデシリアライズすると意図せずスクリプトが実行されてしまう

## 脆弱性が生まれる原因

- 外部からの信頼できない入力データをもとにデシリアライズ処理を行う

## 対策

- シリアライズ形式ではなくJSON形式によりデーターを受け渡す
- クッキーやhiddenパラメーターではなく、セッション変数に書き換えできない形でシリアライズ形式のデータを受け渡す
- HMACなどの改ざん検知の仕組みを導入して改竄されていないことを確認

# 4.14.3 XML外部実態参照

## 概要

- XMLには外部ファイルの内容を取り込む外部実態参照という機能がある
- XMLを外から受け取るプログラムは外部実態参照の形でWebサーバー内部のファイルなどを不正に読み取られる可能性がある → 外部実態参照脆弱性および外部実体参照脆弱性攻撃

## 攻撃手法と影響

### 外部実態参照とは

- XMLの実態はXMLの文書型定義中で以下のように宣言する
    
    ```xml
    <! DOCTYPE foo [
    <! ENTITY greeting "こんにちは">
    <! ENTITY external-file SYSTEM "external.txt">
    ]>
    ```
    
- このようにして宣言した実態はXML文書内で`&greeting`, `&external-flie`という形で参照する
    
    ```xml
    <foo>
    	<hello>&greeting;</hello>
    	<ext>&external-file;</ext>
    </foo>
    ```
    

### 攻撃

- 前提
    - 下記のようなxmlファイルを受け取り、nameとaddressから個人情報を登録するサーバースクリプトがあるとする
    
    ```xml
    <?xml version="1.0" encoding="ut-8" ?>
    <user>
    	<name>安全太郎</name>
    	<address>東京都港区</address>
    </user>
    ```
    
- 攻撃
    
    ```xml
    <?xml version="1.0" encoding="utf-8" ?>
    <!DOCTYPE foo [
    <!ENTITY hosts SYSTEM "/etc/hosts">
    ]>
    <user>
      <name>安全太郎</name>
      <address>&hosts;</address>
    </user>
    
    ```
    
    `/etc/hosts` の中身が読み取られてしまう
    
    !xml脆弱.png
    

## 脆弱性が生まれる原因

XMLの外部実態参照はXMLがもともと持つ機能であり、プログラムにコーディング上のバグがあるというものではない

## 対策

- XMLの代わりにJSONを用いる
- libxml2のバージョン2.9以降を用いる(PHP)
    - 2.9以降で外部実態参照を停止する設定となっている。
    - PHPに限らず、XMLを利用する場合は外部実態参照を停止する設定方法がないか確認するのが良さそう
