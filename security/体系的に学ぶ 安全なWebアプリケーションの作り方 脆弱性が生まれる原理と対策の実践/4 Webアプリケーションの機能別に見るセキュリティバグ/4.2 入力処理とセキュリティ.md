## Webアプリケーションの入力では何をするか

- HTTPリクエストとして渡されるパラメータはGET, POST, クッキーなどがある。本書ではこれらの値の受付時の処理を「入力処理」と呼ぶ
- 入力処理では以下の処理を行う
    1. 文字エンコーディングの妥当性検証
    2. 文字エンコーディングの変換(必要な場合のみ)
    3. 入力値(パラメーター文字列)の妥当性検証

## 入力値の検証

- 目的
    - 入力値の間違いを早期に発見して再入力を促すことにより、ユーザビリティを向上する
    - 間違った処理を継続することによるデータの不整合などを防ぎ、システムの信頼性を向上させる
- セキュリティと入力値の検証
    - 入力値の検証の主目的はセキュリティのためではないが、役に立つ場合もある
        - Ex.
            - SQLインジェクション対策が漏れていたパラメーターがあるが、英数字のみ許可していたので実害にはならない
            - PHPのバイナリセーフでない関数を使っているが、入力段階で制御文字をチェックしているので実害には至らない

### バイナリセーフという考え方とヌルバイト攻撃

- バイナリセーフとは
    - 入力値がどんなバイト列であっても正しく扱えることを意味する
    - 典型的には値ゼロのバイト(ヌルバイト)が現れても正しく処理できることを指す
        - C言語で開発されたPHPなどの言語ではヌルバイトを文字列の終端としてそれ以降を切り詰めてしまう関数がある
            
            → バイナリセーフでない関数
            
- ヌルバイト攻撃
    - ヌルバイトを文字列の終端として扱ってしまうバイナリセーフでないものに対する攻撃
    - Ex.
        - サーバー側
            - クエリパラメーター`p`の値が半角英数字が検証。Falseであればアラートを出す
            
            ```php
            <body>
            <?php
            	$p = $_GET['p'];
            	if (ereg('^[0-9]+$', $p) === FALSE) {
            		die('整数値を入力してください');
            	}
            	echo $p;
            ?>
            </body> 
            ```
            
        - リクエスト(ヌルバイト攻撃 XSS)
            - `http://example.jp/42/42-002.php?p=1%00<script>alert('XSS')</script>`
                - 結果
                    - アラートは出ず、XSSが実行される
                    - ereg関数はバイナリセーフではない関数であるため、ヌルバイトである`%00`で文字が終わっていると考え、`1`のみ評価されて検証が通ってしまった

### 入力値検証の基準はアプリケーション要件

- 入力段階で不正な入力をチェックすればOKとはならない
    - 入力値検証の基準はアプリケーションなので、極端な話「すべての文字を許容する」という仕様の場合は何も防げない

#### 制御文字のチェック

- 「すべての文字を許容する」項目でも検証できる
    - 制御文字とは改行やタブなど普段表示されることのない、ASCIIコード0x20未満および0x7F(DELETE)の文字のこと
- 1行のinput(typeがtextあるいはpassword)のものは通常の入力方法では制御文字が入力できないが、textarea要素の場合は改行やタブを含められるため、タブなどを許容するかを仕様として決定する

#### 文字数のチェック

- 最大文字数をチェックしておくとセキュリティ上の保険的対策になる
    - SQLインジェクション脆弱性があっても文字数が限られているので、攻撃には至らないなど

#### 数値の最小値・最大値のチェック

- 入力された値に応じてメモリを確保するアプリケーションであれば、大きな数字を指定すればメモリ使用量が過大になりクラッシュする可能性も
    - → Dos攻撃
