# 4.6.1 セッションハイジャックの原因と影響

- 第三者がセッションIDを知るための手段は、以下の3種類
    1. セッションIDの推測
    2. セッションIDの盗み出し
    3. セッションIDの強制
- セッションハイジャックの影響
    - 利用者の重要情報の閲覧
    - 利用者の持つ権限での操作
    - 利用者のIDによるメール、ブログなどへの投稿など

# 4.6.2 推測可能なセッションID

## 概要

- セッションIDの生成規則に問題があると、利用者のセッションIDが推測され、セッションハイジャックに悪用される可能性がある
- セッションIDによる脆弱性を作り込まないためには、実績のある言語やミドルウェアが提供するセッション管理機構を使用すること

## 攻撃手法と影響

- 攻撃手法
    - 対象アプリケーションからセッションIDを集める → セッションIDの規則性の仮説を立てる → 推測したセッションIDを対象アプリケーションで試す
    - ありがちなセッションID生成方法
        - ユーザーIDやIPアドレス、日時、乱数といった情報を単独あるいは組み合わせて、それを元にエンコードやハッシュをしてIDを生成
        - ユーザーIDや日時は外部から推測可能であるため、脆弱性の原因になりやすい

## 脆弱性が生まれる原因 と対策

- 原因
    - アプリケーション側でセッション管理を自作しているのが脆弱性の原因を作っている
- 対策
    - 現実的で効果的な対策はWebアプリケーションが備えるセッション管理機構を使うこと
    - どうしても自作しなければならない場合は、暗号論的擬似乱数生成器をもとに、十分な桁数のセッションIDを生成する

# 4.6.3 URL埋め込みのセッションID

- セッションIDをクッキーに保存せず、URLに埋め込ませる方法がある
    - Refererヘッダを経由して、セッションIDが外部に老齢し、なりすましの原因となる場合がある

## 攻撃のシナリオ

- 利用者がリンクを作成できるWebメール、掲示板、ブログSNSを通じて外部から攻撃を仕掛けられる

![IMG_0881.jpeg](attachment:74da5e68-81d1-4f0b-8035-ab1eb8b73e0b:IMG_0881.jpeg)

## 対策

- クッキーにセッションIDを保存するように設定する

# 4.6.4 セッションIDの固定化

- セッションIDを外部から強制する方法があり、セッションIDの固定化攻撃と呼ばれる
- 以下の手順で行われる
    1. セッションIDを入手する
    2. 被害者に対して、1で入手したセッションIDを強制する
    3. 被害者は標的アプリケーションにログインする
    4. 攻撃者は、被害者に強制したセッションIDを使って標的アプリケーションにアクセスする

## 攻撃手法と影響

- ログインページ(http://example.jp/463/46-010.php)に対し、以下のURLによりログインするように誘導する
    
    ```
    http://example.jp/463/46-010.php?PHPSESSID=ABC
    ```
    
- 上記URL経由でアクセスすると、`PHPSESSID=ABC`が有効雨となり、利用者情報はこのセッションに蓄積される。
- 攻撃者は罠に引っかかった利用者がログインしたタイミングを見計らって、被害者の個人情報ページ(http://example.jp/463/46-012.php?PHPSESSID=ABC)により個人情報を閲覧する

## ログイン前のセッションIDの固定化攻撃

- ログイン前のページであっても、セッション変数を使用していると同攻撃が成立する
    - 入力フォーム等の値をセッションに保存しているとき

## クッキーのみにセッションIDを保存するサイトのセッションIDの固定化

- クッキーにセッションIDを外部から設定することはないが、ブラウザやWebアプリケーションに脆弱性があれば、可能になる
- 脆弱性の例
    - クッキーモンスターバグ
    - クロスサイト・スクリプティング脆弱性
    - HTTPヘッダ・インジェクション脆弱性

## 対策

- セッションIDを外部から固定化する方法が多様であるため、固定化自体を防ぐのは現実的ではない
- 対策として「認証後にセッションIDを変更する」　のが良い
